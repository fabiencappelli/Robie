# Premier test de confirmation vocale pour Robie V1.2

Aujourd’hui, objectif volontairement modeste pour Robie V1.2 : obtenir une première boucle fonctionnelle autour des confirmations vocales.

Pas encore de Vosk, pas encore de RapidFuzz, pas encore de Qwen en conditions réelles. L’idée était plutôt de valider une première architecture propre, avec des structures de données stables et une génération de confirmation naturelle.

## Le principe

Pour ce premier test, l’intention (`Intent`) est créée manuellement dans le script.

Autrement dit, on simule le résultat futur du pipeline :

```text
voix
→ transcription
→ compréhension
→ Intent structuré
```

afin de tester uniquement :

```text
Intent
→ question de confirmation naturelle
```

Exemple simulé :

```python
intent = Intent(
    intent="play_audiobook",
    book_id="rdf_04",
    start_mode=StartMode.BEGINNING,
    confidence=0.92,
    raw_text="lis les royaumes de feu tome 4 depuis le début",
)
```

Le programme charge ensuite `library.json`, retrouve le livre correspondant, puis génère une phrase naturelle.

Résultat :

```text
Tu veux écouter Les Royaumes de feu, tome 4, L'île au secret, depuis le début ?
```

## Pourquoi commencer par ça ?

Le piège classique des projets de ce type est de vouloir brancher immédiatement :

- STT
- fuzzy matching
- LLM
- player audio
- orchestration conversationnelle
- sauvegarde d’état
- async
- IPC

… et de finir avec beaucoup de complexité mais rien de réellement fonctionnel.

L’objectif ici est inverse :

- isoler les couches
- valider les contrats de données
- stabiliser les structures internes
- obtenir rapidement un premier comportement observable

Même simple, ce test valide déjà plusieurs éléments importants :

- structure de `library.json`
- modèles Pydantic
- représentation des intentions
- formulation des confirmations
- architecture générale du dialogue

## Structure minimale utilisée

```text
robie/
  classes.py
  library.json
  test_confirmation.py
```

## Architecture visée

Ce test s’inscrit dans une architecture plus large :

```text
Micro
  ↓
Vosk
  ↓
Intent Router
  ├─ intents simples → exécution directe
  └─ intents complexes → Qwen
          ↓
      Intent structuré
          ↓
      Confirmation vocale
          ↓
      mpv
```

Mais pour l’instant, seule cette partie est réellement testée :

```text
Intent structuré
    ↓
confirmation naturelle
```

## Pourquoi le matching fuzzy n’est pas encore implémenté

Le système de recherche locale avec RapidFuzz arrivera dans une étape séparée.

L’idée retenue est :

```text
library.json
    ↓
génération d’un index local
    ↓
matching fuzzy
    ↓
shortlist de candidats
    ↓
Qwen
```

et non :

```text
bibliothèque complète
→ LLM
```

Le LLM ne doit pas être utilisé comme moteur de recherche principal de la bibliothèque.

## Petit win, mais vrai win

Ce genre d’étape paraît minuscule vu de loin.

Mais dans un projet personnel long, surtout sur Raspberry Pi avec plusieurs briques IA et audio, obtenir rapidement un comportement visible et cohérent est extrêmement important.

Même si le système ne “comprend” encore rien, Robie possède déjà :

- une bibliothèque structurée
- des types stables
- des intentions représentables
- une première voix conversationnelle cohérente

C’est une vraie première pierre pour la V1.2.
