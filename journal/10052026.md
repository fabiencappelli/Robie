# Robie V1.2 — Architecture et principes établis

## Objectif de la V1.2

La V1.2 doit permettre :

- reconnaissance d’intention
- confirmation vocale
- lancement d’audiobooks
- reprise à la dernière position
- sauvegarde de progression
- contrôle vocal simple pendant la lecture
- écoute continue pendant la lecture

Fonctionnalités visées :

```text
- “Lis Les Royaumes de feu tome 4”
- “Depuis le début”
- “Reprends”
- “Stop”
- “Pause”
- “Plus fort”
- “Moins fort”
```

---

# Architecture générale

## Pipeline principal

```text
Micro
  ↓
Vosk (STT)
  ↓
Intent Router
  ├─ intents simples → exécution directe
  └─ intents complexes → Qwen
          ↓
      Intent Pydantic
          ↓
      Confirmation vocale
          ↓
      mpv
```

---

# Technologies retenues

## STT

### Vosk

Utilisé pour :

- transcription locale
- faible latence
- fonctionnement offline
- Raspberry Pi compatible

---

## Compréhension complexe

### Qwen

Utilisé uniquement pour :

- compréhension sémantique complexe
- désambiguïsation
- formulation des confirmations
- corrections partielles après confirmation

Pas utilisé pour :

- stop
- pause
- volume
- commandes triviales

---

## Orchestration

### LangGraph

Utilisé pour :

- gérer les états conversationnels
- confirmations
- corrections
- intentions en attente
- mémoire courte de session

---

## Audio playback

### mpv

Choisi car :

- support du seek précis
- contrôle IPC JSON
- pause/reprise
- contrôle volume
- récupération de position courante
- adapté aux audiobooks

### `aplay` rejeté

Raison :

- pas adapté à la reprise à un timestamp précis
- pas conçu comme moteur interactif de lecture audiobook

# Pydantic

Les intentions seront représentées via modèles structurés.

Exemple :

```python
class Intent(BaseModel):
    intent: Literal[
        "play_audiobook",
        "resume_audiobook",
        "stop",
        "pause",
        "resume_playback",
        "volume_up",
        "volume_down",
        "unknown",
    ]

    book_id: Optional[str]
    start_mode: Optional[
        Literal["beginning", "last_position"]
    ]

    confidence: float
    confirmation_question: Optional[str]
```

# Sauvegarde des positions

## Séparation stricte

Il faut préserver le fichier library d'une réécriture constante, en cas de plantage de Robie par ex. D'où l'écriture de positions sur un fichier différent. Cela ouvre aussi la porte à un fichier de position par personne.

### `library.json`

Décrit :

- les livres
- les métadonnées
- les chemins

---

### `positions.json`

Décrit :

- la progression de lecture

Exemple :

```json
{
  "rdf_04": {
    "position_seconds": 4821.7,
    "updated_at": "2026-05-10T12:00:00",
    "completed": false
  }
}
```

---

# Sauvegarde automatique

La position doit être sauvegardée :

```text
- au stop
- régulièrement pendant la lecture
```

Pour éviter la perte de progression.

# Recherche de livres

## Important

Qwen ne doit PAS recevoir toute la bibliothèque.

Architecture retenue :

```text
transcription
  ↓
matching local
  ↓
shortlist
  ↓
Qwen
```

---

# Matching fuzzy

## RapidFuzz

Utilisé pour :

- erreurs STT
- variations de formulation
- approximations
- pluriels/singuliers
- ordre des mots

---

# Principe du matching

## Génération d’un index de recherche

Exemple :

```text
"L'île au secret"
"Les Royaumes de feu tome 4"
"Royaumes de feu 4"
"Tome 4 des Royaumes de feu"
"Royaumes de feu tome quatre"
```

Chaque entrée pointe vers :

```text
book_id
```

Pour l'instant on va générer cet index au démarrage, mais il faudra penser à regénérer en cas de changement de la bibliothèque. Soit automatisé, soit sur commande.

---

# Normalisation retenue

## Normalisation légère uniquement

Conserver les chiffres (par exemple pour les tomes) écrits en toutes lettres, car les erreurs phonétiques de Vosk doivent rester comparables.

Donc :

```text
- lowercase
- suppression accents
- suppression ponctuation
```

Mais PAS conversion agressive :

```text
quatre → 4
```

Pour chaque tome ce sont donc les chiffres en toutes lettres qui devront être ajoutés à l’index.

---

# Rôle exact de RapidFuzz

RapidFuzz :

- retrouve les candidats plausibles
- ne comprend pas l’intention

Qwen :

- comprend ce que l’utilisateur veut faire
- choisit entre candidats
- formule confirmations

---

# Confirmations

## Point important établi

Les confirmations ne sont PAS binaires.

Exemple :

```text
Robie :
“Tu veux écouter L’île au secret depuis le début ?”

Enfant :
“Non, reprends là où ça en était.”
```

Ici :

- le livre est correct
- seul `start_mode` est corrigé

# Principe architectural central

Le point majeur établi :

```text
Le LLM ne doit pas être le moteur de recherche de la bibliothèque.
```

Le système correct :

```text
matching local
→ shortlist
→ compréhension LLM
```

et non :

```text
bibliothèque complète
→ LLM
```
